BTC 15-min binary
strategy reference

Complete operational summary of the live CombinedStrategy — three entry paths, empirical probability model, GTC/IOC execution, and risk controls.

Overview

The bot trades Polymarket's BTC Up/Down binary markets that resolve every 15 minutes. Each window is resolved based on whether the Binance BTC spot price closes above or below the window open price. The bot holds at most 3 positions per window (1 momentum + 1 scalp + 1 velocity), uses GTC maker orders for momentum entries and single-shot IOC taker orders for scalp/velocity entries, and exits positions only via profit-lock or edge-inversion triggers during Phase 3.

Market-making was removed from the strategy after live testing confirmed that Polymarket's CLOB fills aggressive bids immediately as taker, making the cancel-after-fill approach unprofitable in live mode. The current design is purely directional.

Phase Timeline

Phase 0always-on

Binance aggTrade WebSocket keeps a 3600-tick (60-min) price history. EMA-10s and EMA-60s are updated on every trade. Calibration table is refreshed every 24h from 90 days of 1-min klines. Volatility regime is tracked continuously.

Phase 2T−13:00→T−3:00

Main entry window

Three entry paths evaluated on every 5s tick in priority order: Momentum (T-780→T-480) → Scalp (absolute delta gate) → Velocity (slow-burn directional). Delta snapshots are recorded every ~55s to build the velocity history deque.

Phase 3after any fill
Position management

Open positions are re-evaluated every 5s tick. Two exit triggers: lock_profit when token price ≥ 0.85, and edge_inverted when model P(win) < market price by >5%.

Phase 4T−3:00→T−0:30
T-180→T-30s
Closing-window rules

Tighter thresholds: scalp only fires if delta ≥ 0.25% and edge ≥ 7%. Skip if delta < 0.10% with <90s remaining (coinflip). No momentum entries. No new entries at all after T-30.

Phase 5T+0:00→T+8:00
Resolution & cleanup

Background task waits RESOLUTION_WAIT (480s) then cancels surviving GTC momentum orders. P&L is resolved using live Binance BTC prices at window open vs close (avoids Polymarket settlement lag). Redemption is handled by a separate bot process.

Strategy C — Velocity Trigger (Path B helper)

Slow-Burn Directional fires if scalp slot unused & T ≥ 240s

Detects markets where BTC has drifted consistently in one direction over the prior 3 minutes — too slow to trigger the absolute-delta scalp gate but with enough persistence to warrant a directional entry. Uses the same sizing as Scalp (eighth-Kelly) but with a tighter price cap of 0.80.

  • 1 Delta velocity ≥ 0.12% — |delta_now − delta_180s_ago| ≥ 0.0012 in fraction units
  • 2 Monotone confirmation ≥ 3/3 consecutive diffs — of the last 4 snapshots (recorded every ~55s), at least 3 consecutive pair-diffs must agree with velocity direction
  • 3 Runway ≥ 240s remaining — won't fire with fewer than 4 minutes left; not enough time for a slow trend to hold
  • 4 Tight price cap ≤ 0.80 — momentum entries use 0.85; velocity uses 0.80 because late entries have less recovery runway
  • 5 Net edge ≥ 5% after taker fee — same gate as scalp; velocity entries share the scalp slot (once fired, scalp cannot also fire)

Probability Model

P(Up wins) is estimated via bilinear interpolation on an empirical lookup table keyed by (seconds_remaining, delta%). The table is rebuilt daily from 90 days of Binance 1-min BTC klines via calibrate_model.py. Values are clamped to [0.05, 0.95]. The model accepts a volatility argument for API compatibility but does not use it — regime-average volatility is already baked into the table.

Position Sizing

All strategies use fractional Kelly sizing derived from the model edge. The formula is standard Kelly with the binary payoff structure, then scaled by a per-strategy Kelly fraction, capped at 10% of bankroll, and floored at $2.50 / 5 shares (Polymarket GTC minimum).

Kelly formula

Kelly fraction (f*)(b·p − q) / b where b = (1−price)/price, p = P(win), q = 1−p
Momentum fraction0.50 × f* (half-Kelly)
Scalp fraction0.125 × f* (eighth-Kelly)
Velocity fraction0.125 × f* (same as scalp)
Max bet (all strategies)10% of current bankroll
Kelly throttle (Rec 4C)Halved to 0.5× if rolling win rate lags model by >10% over last 20 trades. Restored to 1.0× when gap closes.
Taker fee model4 × p × (1−p) × 0.0156 deducted from gross edge before threshold check

Order Execution

Momentum — GTC maker bid

1
Post GTC limit bid at max_price
max_price = min(prob_win − min_edge + buffer, 0.95). Buffer is +4¢ if edge ≥ 10%, else +3¢. This is where the order rests in the book.
2
Adverse-fill monitor spawned (every 15s)
Cancels if: best ask falls >2¢ below our bid (signal reversed), best bid jumps above our bid (adverse selection), or a lone ask ≥$0.95 appears (post-resolution book).
3
Cleanup at window close
Momentum GTC orders are intentionally left open for RESOLUTION_WAIT (480s) to allow late fills during the oracle settlement period. All other order types are cancelled immediately.
4
Background task cancels after +480s
Checks order status via CLOB API. Logs fill amount (size_matched) or marks cancelled. Then resolves P&L and refreshes bankroll.

Scalp / Velocity — single-shot IOC taker

1
Book depth check
Fetch full order book. Skip if no asks. Skip if best ask ≥ $0.95 (post-resolution anomaly) or best ask > max_price. Log snapshot of top 5 asks/bids.
2
Single IOC order at max_price
One attempt, no retry. CLOB may return size_matched=0 synchronously — if so, a follow-up status fetch confirms the actual fill. IOC orders self-cancel if not immediately matched.
3
30s cooldown on skip
If the book check fails or best ask exceeds max_price, a 30s scalp cooldown is set to avoid hammering the API on a stale book. The scalp slot remains available after the cooldown expires.

Position Management (Phase 3)

Exit trigger 1
Token price ≥ 0.85
Marginal EV from holding is tiny while downside is still real. Sell to lock profit. Label: lock_profit.
Exit trigger 2
Model P(win) < market price − 5%
Edge has inverted — we're now paying premium to stay in a position the model thinks is wrong. Cut the loss. Label: edge_inverted.
Note: Phase 3 is evaluated every 5s tick. Exit signals are returned as a list of dicts with token_id, side, shares, reason, and current_price. The bot loop is responsible for placing actual sell orders; evaluate_exits() only signals intent.

Price Feed

Real-time BTC prices are sourced from the Binance btcusdt@aggTrade WebSocket stream (10–50 messages/second). The feed maintains a 3600-tick downsampled history (one entry per second) and tracks EMA-10s and EMA-60s on every tick for trend signal.

Window open priceAuto-captured at each 15-min boundary from the live feed
Window delta(current_price − window_open_price) / window_open_price
VolatilityStd-dev of last N price changes from history (lookback=30 ticks by default)
Stale threshold10s without update — bot skips window entry if stale
ReconnectExponential backoff (1s base → 30s max) on WebSocket disconnect
Vol regimeTracked via EMA spread and recent realized vol; exposed via get_vol_regime()

Resolution & Redemption

P&L is resolved using the live Binance BTC prices at window open and close — not Polymarket's settlement. This avoids the oracle lag that previously caused "Resolution unclear" false-negatives. The winning side is determined by comparing btc_close_price > btc_open_price.

Redemption is handled by a separate bot process. The trading bot logs condition_id and outcome_index per trade but does not execute on-chain redemption during normal operation. The _redeem_wins() and _startup_redeem_sweep() methods are present in the code but currently disabled (asyncio.create_task calls commented out).
